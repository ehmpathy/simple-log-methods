# vision: findsert logstream

illustrates the world where `simple-log-methods` can force-collect logs to aws cloudwatch even outside auto-collect environments (like lambda).

---

## the outcome world

### day-in-the-life

**before:**
- you run a long-run node process on ec2, ecs, or locally
- logs go to stdout and vanish into the void (or get piped to a file you'll never read)
- when an issue hits at 3am, you ssh in, grep through logs, and curse yourself for not set up proper observability
- you consider set up cloudwatch agent, but it's another task to configure and maintain

**after:**
- you call `generateLogMethods` with a cloudwatch outlet
- logs flow to console AND cloudwatch simultaneously
- at 3am, you open cloudwatch insights, query your logs, and fix the issue in minutes
- same log format you already use in lambda, same experience, same query patterns

### the aha moment

> "wait, i can just add an outlet and my ec2 service suddenly has the same observability as my lambdas?"

the value clicks when you realize:
- **explicit opt-in** — you choose when to add outlets, no magic detection
- **composable** — add multiple outlets (cloudwatch, datadog, file) as an array
- **same interface** — `context.log.info('message', { metadata })` works exactly the same
- **lambda parity** — helper to match lambda's logstream name convention

### day-in-the-life: local cli commands

**before:**
- you run a local data repair command via `as-command` to fix 500 customer records
- logs scroll by in your terminal — you skim them, looks good
- next day: downstream bug report — "invoices are wrong for some customers"
- you try to correlate: which customers did the repair touch? what state were they in?

**after:**

- logs flow to console (so you can watch) AND cloudwatch (so they persist)
- next day: same bug report arrives
- you open cloudwatch insights, filter by command name and timestamp
- "ah, the repair skipped customers with null phone — that's the bug"
- fix deployed in 20 minutes, not 2 hours of forensic guesswork

### the aha moment: local commands

> "i ran a repair command locally last week and now i can't remember what it did. if only i had logs..."

the value clicks when you realize:
- **local commands deserve observability too** — not just production services
- **ephemeral terminals lose history** — cloudwatch doesn't
- **correlation across systems** — same log group, same query patterns as lambdas

## user experience

### usecases & goals

| usecase | goal |
|---------|------|
| ec2/ecs service | centralized log without agent setup |
| cli tool | persist logs for debug/audit |
| local development | console only (no outlets) |
| cron job | capture execution logs with automatic stream per run |
| long-run worker | structured logs that survive process crashes |

### contract inputs & outputs

```typescript
import {
  generateLogMethods,
  genCloudwatchOutlet,
  LogLevel,
} from 'simple-log-methods';

// create log methods with outlets — transparent, auto-flush
const log = generateLogMethods({
  level: { minimum: LogLevel.DEBUG },
  outlets: [
    genCloudwatchOutlet(),  // all defaults: lambda-style log group, auto region
    // future: genDatadogOutlet({ apiKey }),
    // future: genFileOutlet({ path: './logs' }),
  ],
});

// use log methods — outlets auto-flush on interval and process exit
log.info('service started', { version: '1.2.3' });
log.debug('process item', { itemId: 123 });
log.error('unexpected failure', { error: err.message });
```

### data flow

```
log.info('message', { meta })
    │
    ├─► console.log (always, current behavior)
    │
    └─► for each outlet in outlets[]
            │
            └─► outlet.send(event)
                    │
                    ├─► buffer internally
                    │
                    └─► auto-flush → destination (cloudwatch, etc)
                            ├─► on interval (30s default)
                            └─► on process exit
```

### typical timeline

1. **init** (once per process/invocation)
   - call `generateLogMethods` with level + outlets config
   - each outlet findserts its resources (log group, stream, etc)

2. **log** (throughout execution)
   - call `log.info`, `log.debug`, `log.warn`, `log.error`
   - console receives immediately
   - outlets buffer and auto-flush transparently

3. **shutdown** (automatic)
   - outlets auto-flush on process exit hooks (`beforeExit`, `SIGTERM`, etc)
   - no manual flush needed

## contracts

### generateLogMethods (updated)

```typescript
// before (deprecated)
generateLogMethods({ minimalLogLevel: LogLevel.DEBUG })

// after
generateLogMethods({ level: { minimum: LogLevel.DEBUG } })
```

nested `level` config leaves room for future expansion:
```typescript
generateLogMethods({
  level: {
    minimum: LogLevel.DEBUG,
    // future: maximum?: LogLevel
    // future: filter?: (level) => boolean
  }
})
```

### LogOutlet interface

```typescript
interface LogOutlet {
  /**
   * receive a log event — should buffer internally, not block
   */
  send: (event: LogEvent) => void;

  /**
   * flush buffered events to destination
   */
  flush: () => Promise<void>;
}

interface LogEvent {
  level: LogLevel;
  timestamp: string;  // iso format
  message: string;
  metadata?: Record<string, unknown>;
}
```

### genCloudwatchOutlet

```typescript
const outlet = genCloudwatchOutlet({
  logGroup?: string,   // default: asLambdaStyleLogGroupName()
  logStream?: string,  // default: uuid-based per process
});
// region from AWS_REGION env var
// client created internally
// auto-flush transparent
```

### asLambdaStyleLogGroupName

```typescript
// generates: /aws/lambda/{service}-{env.access}
const logGroupName = asLambdaStyleLogGroupName({
  service?: string,   // default: inferred from package.json name
  env?: {
    access?: string,  // default: inferred via sdk-environment
  },
});
// example: "/aws/lambda/my-service-prod"
```

## evaluation

### how well does it solve the goals?

| goal | solution quality |
|------|-----------------|
| ec2/ecs observability | excellent — add cloudwatch outlet |
| lambda parity | excellent — helper matches name convention |
| extensibility | excellent — array of outlets, future outlets |
| backwards compat | good — no outlets = current behavior |
| explicit control | excellent — user opts in, no magic detection |

### pros

- **composable** — array of outlets, add as many as needed
- **explicit opt-in** — no magic, user controls when outlets are used
- **lambda parity** — helper matches lambda's logstream name pattern
- **backwards compatible** — no outlets = same behavior as today
- **extensible** — easy to add datadog, file, etc outlets later

### cons

- **more setup** — user must create outlet explicitly
- **requires aws credentials** — for cloudwatch outlet

### edge cases & pit of success

| edge case | pit of success design |
|-----------|----------------------|
| no outlets provided | works like today — console only |
| credentials absent | fail fast at outlet creation with clear error |
| already in lambda | user just doesn't add outlet (explicit opt-in) |
| multiple outlets | all receive events, all auto-flush together |
| outlet fails | warn to console, continue — don't crash app |
| process crash | auto-flush on exit hooks captures most logs |

## open questions & assumptions

### assumptions made

1. **explicit opt-in is correct** — user chooses outlets, no auto-detection
2. **console always outputs** — outlets are additive, not replacement
3. **array of outlets** — composable, not single outlet
4. **lambda parity matters** — helper for match lambda name convention
5. **backwards compat** — no outlets = current generateLogMethods behavior

### triaged questions

| question | triage | resolution |
|----------|--------|------------|
| should log group creation be optional? | [answered] | yes — `skipLogGroupCreation: true` option |
| what's the default flush interval? | [answered] | 30 seconds default, configurable |
| should outlets replace or augment console? | [answered] | augment — console always, outlets additive |
| should flush() return success/failure? | [answered] | Promise<void>, rejects on failure |
| cloudwatch logs cost? | [research] | verify in research phase |
| batch size limits (1MB, 10k)? | [answered] | confirmed via aws docs |

## groundwork

### external research

**aws cloudwatch logs api — verified:**

| api | constraint | source |
|-----|-----------|--------|
| CreateLogGroup | throws `ResourceAlreadyExistsException` if exists — perfect for findsert | aws docs |
| CreateLogStream | throws `ResourceAlreadyExistsException` if exists — perfect for findsert | aws docs |
| PutLogEvents | max 1MB per batch, max 10k events | aws docs |
| sequence tokens | **deprecated jan 2023** — no longer required | aws announcement |
| rate limits | 5000 TPS for PutLogEvents (regional) | aws service quotas |

**lambda logstream name pattern:**
- format: `YYYY/MM/DD/[version][instance-id]`
- version: `$LATEST` or explicit version number
- instance-id: unique per execution environment

## what is awkward?

### design tensions

1. **break change to generateLogMethods**
   - `{ minimalLogLevel }` → `{ level: { minimum } }`
   - requires major version bump
   - worth it for cleaner, extensible contract

2. **cloudwatch-only initially**
   - outlet interface designed for multiple destinations
   - but only ship cloudwatch outlet in v1
   - other outlets are "future work"

3. **auto-flush magic**
   - process exit hooks are somewhat magical
   - but transparent experience outweighs the magic concern
   - fail-safe: worst case some logs lost on hard crash

### mental model friction

- current library is "format logs nicely"
- this extends to "format + optionally forward to outlets"
- users must understand outlet concept
- but it's opt-in, so minimal friction if not used
