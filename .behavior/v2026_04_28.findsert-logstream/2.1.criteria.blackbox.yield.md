# blackbox criteria: findsert logstream

experience boundaries for log outlets — what users see, do, and receive.

---

# usecase.1 = log with cloudwatch outlet

given a cloudwatch outlet is configured
  when generateLogMethods is called with the outlet
    then log methods are returned
      sothat users can log messages
  when log.info is called with a message
    then the message appears in console immediately
      sothat users see feedback in terminal
    then the message is sent to cloudwatch
      sothat logs persist for later query
  when log.debug, log.warn, log.error are called
    then each level flows to console and cloudwatch
      sothat all log levels are captured

given logs have been sent to cloudwatch
  when user queries cloudwatch insights
    then logs appear with timestamp, level, message, metadata
      sothat users can filter and correlate

---

# usecase.2 = auto-flush transparency

given a cloudwatch outlet is configured
  when the process exits normally
    then buffered logs are flushed to cloudwatch automatically
      sothat no logs are lost on shutdown
  when the process receives SIGTERM
    then buffered logs are flushed before exit
      sothat graceful shutdown captures all logs
  when the flush interval elapses (default 30s)
    then buffered logs are flushed to cloudwatch
      sothat logs appear within reasonable delay

given a cloudwatch outlet is configured
  when user logs messages
    then no manual flush is required
      sothat the experience is transparent

---

# usecase.3 = lambda-style log group name

given genCloudwatchOutlet is called with no arguments
  when asLambdaStyleLogGroupName is used internally
    then log group name is /aws/lambda/{service}-{env.access}
      sothat logs match lambda convention
  when service is not provided
    then service is inferred from package.json name
      sothat zero config works
  when env.access is not provided
    then env.access is inferred via sdk-environment
      sothat zero config works

given genCloudwatchOutlet is called with explicit logGroup
  when logs are sent
    then logs go to the specified log group
      sothat users can override defaults

---

# usecase.4 = backwards compatibility

given generateLogMethods is called without outlets
  when log methods are used
    then logs go to console only
      sothat extant behavior is preserved
  when no outlets array is provided
    then the library works exactly as before
      sothat no migration is required for console-only usage

---

# usecase.5 = multiple outlets

given multiple outlets are configured
  when log.info is called
    then message flows to console AND all outlets
      sothat users can send to cloudwatch + datadog + file simultaneously
  when one outlet fails
    then the error propagates to the caller
      sothat failures are immediately visible
    then the process crashes if unhandled
      sothat log failures are never silently swallowed

note: fail-fast philosophy — if someone expected logs, we gotta fail hard and fail fast if we can't support it. silent failures in log output lead to lost observability at critical moments.

---

# usecase.6 = log group findsert

given a cloudwatch outlet with a log group that doesn't exist
  when the first log is sent
    then the log group is created automatically
      sothat users don't need to pre-create in AWS console
  when the log group already exists
    then logs are sent without error
      sothat findsert semantics apply

given a cloudwatch outlet with skipLogGroupCreation: true
  when the log group doesn't exist
    then an error is raised
      sothat users who pre-create via terraform get clear failures

---

# usecase.7 = log stream per process

given a cloudwatch outlet is configured
  when the process starts
    then a unique log stream is created (uuid-based)
      sothat each process/invocation has its own stream
  when multiple processes run simultaneously
    then each has a distinct log stream
      sothat logs don't interleave

---

# usecase.8 = credential errors

given AWS credentials are not configured
  when genCloudwatchOutlet is called
    then an error is raised immediately
      sothat users know credentials are required
    then the error message includes guidance
      sothat users can fix the config

given AWS credentials lack cloudwatch permissions
  when the first log is sent
    then an error is raised
      sothat users know IAM permissions are required
